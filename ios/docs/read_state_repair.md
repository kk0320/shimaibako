# 読取状態再確認設計

## 目的

読取結果キャッシュを削除しなくても、500件/2,000件読取の対象抽出が正しく動くようにする。

読取結果キャッシュと検索インデックスは別物として扱う。検索データがあるだけでは読取済みとはみなさない。

## データの区別

読取結果キャッシュ:

- OCR本文
- 文字なし判定
- 読取失敗情報
- 処理日時と言語

検索インデックス:

- OCR結果から作った検索用テキスト
- カテゴリ
- スクショ細分類
- メモ
- タグ
- 日付や種類などの検索用メタデータ

検索インデックス件数が多くても、それは読取済み件数ではない。SearchDocumentだけがある写真は、未読取候補に含める。

## 対象抽出

除外するもの:

- OCR本文あり
- 文字なし判定済み
- 現在処理中
- 進行中ジョブのpending/processing item

候補に含めるもの:

- SearchDocumentだけがある写真
- PhotoIndexRecordだけがある写真
- カテゴリ分類だけがある写真
- メモやタグだけがある写真
- OCRメタデータのない古い空のcompleted状態
- retry可能な失敗状態

## 読取状態を再確認

設定画面の「読取状態を再確認」は、削除ではなく修復である。

行うこと:

- 現在の写真一覧とOCR結果キャッシュを照合する
- OCR本文や文字なし判定があるものは保持する
- 手動分類、不要候補、非表示、整理済み、メモ、タグを保持する
- OCR本文も処理日時もない空のcompleted状態を未読取扱いへ戻す
- staleなprocessing状態を未読取扱いへ戻す
- 無効な0件ジョブや旧全数OCR由来の状態を通常UIから外す

行わないこと:

- 元写真・元動画を削除/変更しない
- OCR本文を削除しない
- 文字なし判定を削除しない
- 検索インデックスを全削除しない
- 手動分類、不要候補、メモ、タグを削除しない
- 写真アプリ側の写真、動画、アルバム、iCloud写真を変更しない

## 旧全数OCR状態

過去の全数OCR実験由来のジョブ状態は、通常UIと対象抽出に影響させない。

- 0/0 completed job は表示しない
- 2,000件を超えるジョブ状態は無効扱いにする
- 全数OCR系の表示や管理導線は通常UIに戻さない
- 保存済みの有効なOCR本文や文字なし判定は保持する

## 検証

DEBUGビルドで対象抽出検証を実行する。

- 検索データのみ写真の対象化テスト
- キャッシュ削除なし500件対象抽出テスト
- キャッシュ削除なし2,000件対象抽出テスト
- 0件対象テスト

証跡は `evidence/batch_ocr_target_selection_validation/` に保存する。
