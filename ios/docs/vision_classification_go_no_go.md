# Vision Classification Go/No-Go

## 結論

P0.6時点では、Vision標準機能をそのまま整理タブ本実装へ入れるのはまだ早い。

ただし、次の限定用途なら本実装候補に進める価値がある。

- スクショ高速判定
- スクショのOCR優先候補化
- レシート/書類の弱い候補提示
- 顔/人物を含む可能性の軽量表示

建物、工事現場、看板、白板、図面、名刺は、手動正解ラベル付きの固定サンプル評価を追加してから判断する。

## P0からP0.6の総括

分かったこと:

- Vision標準機能は速度面では有望。
- `VNDetectDocumentSegmentationRequest` は全件反応しやすく、単独では使えない。
- スクショと書類の分離は有効。
- スクショは書類ではなくOCR優先として扱うのがよい。
- `gatedProbe` によりスクショでは重いVision requestを省略できる。

K Phone P0.6:

| bucket | mode | count | avg ms | finalDocument | ocrPriority |
| --- | --- | ---: | ---: | ---: | ---: |
| スクショ | fullProbe | 20 | 527.8 | 0 | 20 |
| スクショ | gatedProbe | 20 | 0.0 | 0 | 20 |
| 直近 | fullProbe | 100 | 55.1 | 1 | 87 |
| 直近 | gatedProbe | 100 | 7.8 | 1 | 87 |
| スクショ以外 | fullProbe | 20 | 52.8 | 1 | 0 |
| スクショ以外 | gatedProbe | 20 | 49.3 | 1 | 0 |

## 採用する信号

### スクショ

採用する。

- PhotoKitの `photoScreenshot` を優先する。
- スクショには原則として重いVision分類をかけない。
- スクショは「記録・メモ」用途としてOCR優先候補にする。

### OCR優先度

限定採用する。

- スクショは高いOCR優先度にする。
- レシート、名刺、書類、看板、白板はOCR優先度に寄与させる。
- 本番で読取タブに直接投入する前に、ユーザー確認または段階処理を挟む。

### レシート/書類

弱い候補として採用検討する。

- `documentLabelScore`
- `documentVisualScore`
- `receiptScore`

`documentSegmentation` は主信号にしない。

### 人物/顔

候補表示として採用検討する。

- 顔画像は保存しない。
- 顔テンプレートは保存しない。
- 人物識別はしない。
- 「人物を含む可能性」程度に留める。

## 採用しない信号

### documentSegmentation単独判定

採用しない。

P0.5/P0.6ともに全件反応に近く、書類候補の主判定には使えない。使う場合も0.0〜0.1程度の弱い補助にする。

### 画像特徴ベクトル保存

採用しない。

3万枚規模では保存容量、発熱、バッテリー、削除設計の確認が必要。P0.7ではまだ扱わない。

### 外部Core MLモデル

採用しない。

ライセンス、モデルサイズ、端末負荷、保守性が未確認。Vision標準機能と手動正解ラベル評価を先に完了する。

## 初期リリースに入れる候補タグ

候補:

- スクショ
- OCR優先
- レシート候補
- 書類候補
- 人物を含む可能性

これらも断定表示ではなく「候補」として扱う。

## 後回しにするタグ

- 建物
- 工事現場
- 看板
- 白板
- 図面
- 名刺
- 資材・設備
- 車両・重機
- 食べ物
- 風景

理由:

- 今回のK Phoneサンプルに十分な正解例がない。
- Vision標準ラベルだけでは直接拾いにくいものがある。
- 手動正解ラベル付きprecision / recallが必要。

## 整理タブ本実装の最小仕様案

まだ実装しない。

将来の最小仕様:

- しまい箱内だけの仮想タグとして保存する。
- 写真アプリ側のアルバムや元写真・元動画は変更しない。
- 初回はスクショ/OCR優先/書類候補のような低リスクタグだけ扱う。
- ユーザーが候補を修正できる。
- 手動分類を自動分類より優先する。

## ClassificationJob最小仕様案

まだ実装しない。

将来の最小仕様:

- 1ジョブ最大500〜2,000件程度。
- スクショはPhotoKitメタデータだけで高速処理。
- 一般写真は小分けで処理。
- 端末状態が悪い場合は一時停止する。
- 画像本体、サムネイル本体、顔画像、顔テンプレートは保存しない。
- 結果は候補タグとしてアプリ内DBに保存する。

## OCR候補連携の最小仕様案

まだ本連携しない。

将来の最小仕様:

- OCR優先度の高い候補を読取タブに表示する。
- 自動で大量OCRへ流さない。
- ユーザーが20/50/100/500/2,000件の段階読取を選ぶ。
- BatchOCRJobの既存上限と中断/再開設計を使う。

## Go/No-Go観点

| 観点 | 判断 |
| --- | --- |
| 速度 | スクショgatedはGo。一般写真は小分け前提 |
| 精度 | スクショ/OCR優先はGo寄り。一般分類は追加評価が必要 |
| 誤検出 | documentSegmentation単独はNo-Go |
| スクショ処理 | Go |
| 書類処理 | 弱い候補ならGo、断定はNo-Go |
| 建物/工事現場/看板 | No-Go、固定サンプル不足 |
| OCR優先度 | Go寄り、読取タブへの自動投入はまだNo-Go |
| 発熱 | 小規模ベンチでは問題なし。大量ClassificationJobは未評価 |
| UI影響 | DEBUG限定では問題なし。本実装UIは未実装 |
| 実装複雑度 | スクショgatedは低い。一般分類ジョブは中 |
| プライバシー | 画像保存なしならGo |

## P0.9後の3段階Go/No-Go

### Engine Go

合成画像で判断してよい範囲。

Go条件:

- File-based benchmarkが動く。
- score計算が安定している。
- semantic assertionのPASS/FAILが出る。
- JSON/CSV/Markdown evidenceが出る。
- metadataAwareスクショ高速パスが退行しない。
- fixture画像がRelease targetや.appへ混入しない。

Engine Goは、分類エンジンの処理経路が動くという意味に留める。ユーザー向け自動フォルダ公開の判断ではない。

### Candidate Go

ライセンス確認済み実写真で判断する範囲。

Go条件:

- カテゴリ信号が分離できる。
- 致命的な誤分類パターンがない。
- 閾値調整の見込みがある。
- ライセンス、再配布、個人情報、商標の確認が済んでいる。

Candidate Goでも、建物や工事現場などのユーザー向け断定表示はまだ行わない。

### Product Go

自分で撮影した写真、許可済み現場写真など、in-domain holdoutだけで判断する。

Product Go対象:

- 建物フォルダを公開する。
- 工事現場フォルダを公開する。
- 車両・重機フォルダを公開する。
- 資材・設備フォルダを公開する。

合成画像や一般公開画像だけでProduct Goにしない。

## P0.9の暫定判断

Go:

- スクショ高速判定。
- スクショをOCR優先候補にする。
- File URL入力のDEBUGベンチを追加する。
- 合成fixtureによるscore経路の退行検出。

No-Go:

- 整理タブ本実装。
- 5タブ化本実装。
- ClassificationJob本実装。
- 建物/工事現場/看板/白板/図面/名刺/レシートを一般UIで断定表示。
- 本番分類DB保存。
- 写真タブカテゴリへの本反映。
- 読取タブOCR候補への本連携。
- `main` へのmerge。

## 次に必要なこと

- DEBUG UIで20件以上の手動正解ラベルを付ける。
- 建物、工事現場、看板、白板、図面、名刺、レシートの固定サンプルを意識して評価する。
- precision / recallを見て、初期タグの範囲を再判断する。
- その後に `feature/local-image-classification` を作るか判断する。

## P0.8で追加した判断材料

P0.8では、Go/No-Go判断のために手動ラベル付けと評価再実行の導線を整える。これはDEBUG限定であり、整理タブ本実装、ClassificationJob、本番分類DB保存、読取タブ連携はまだ行わない。

追加したレビュー対象:

- 直近20件、直近50件、直近100件
- スクショ候補20件
- スクショ以外20件
- OCR優先候補20件
- 建物候補20件
- 工事現場候補20件
- 看板候補20件
- 白板候補20件
- 書類候補20件

評価再実行では、手動ラベル済みデータだけを対象に `support`、`TP`、`FP`、`FN`、`precision`、`recall` を出す。ラベル済み件数が0件の場合は評価不可として扱う。

P0.8後にGoへ寄せられる条件:

- 少なくとも20〜50件の手動ラベルがある。
- スクショ/OCR必要のprecisionが高い。
- 書類、看板、建物、工事現場の誤検出傾向を具体的に説明できる。
- 建物/工事現場/看板/白板などの弱いカテゴリを、初期UIで断定表示しない方針が明確である。
- 画像本体、サムネイル本体、顔画像、顔テンプレートを保存しないまま評価できている。

P0.8後もNo-Goに留める条件:

- 手動ラベルが0件またはごく少数。
- supportが少なく、precision / recallをカテゴリ判断に使えない。
- documentSegmentationの過検出を補正できていない。
- 建物/工事現場など、ユーザーが期待する粒度で見分けられる根拠がない。
- 本番分類DBや写真タブに反映した時の手動修正導線が未設計。

## 安全方針

- 元写真・元動画は削除・変更しない。
- PhotoKit書き込み/削除APIは使わない。
- 画像本体は保存しない。
- サムネイル本体は保存しない。
- 顔画像/顔テンプレートは保存しない。
- 本番分類DB、写真タブ、読取タブ、BatchOCRJobにはまだ反映しない。
