# UI状態分離計画

## 目的

28,000件規模の写真ライブラリで、検索インデックス進捗、件数チップ、写真グリッド、サムネイル取得が互いに巻き込まれて再描画される状態を避ける。

## 分離する4つの経路

### 検索インデックス進捗

- 旧JSON移行や検索文書バックフィルの進捗は `IndexProgressStore` へ流す
- 写真画面と設定画面の進捗カードだけが購読する
- 500件単位の進捗を出し、1件ごとのUI更新はしない
- 進捗更新で写真グリッド全体を再生成しない

### 件数チップ

- 表示状態、カテゴリ、スクショ細分類の件数は `FilterCountsSnapshot` として扱う
- 集計はSQLiteで行い、SwiftUIのbody内で全件 `.filter { }.count` をしない
- 集計中は `--` を表示し、未計算とゼロ件を区別する
- カテゴリの「すべて」は現在の表示状態スコープに合わせる

### 写真グリッド

- グリッドへ渡すのはSQLiteから取得したIDページである
- 初期表示は200件、追加表示も200件単位にする
- 検索語、表示状態、カテゴリ、スクショ細分類はDB条件に入れる
- 古い検索結果や古いページ取得結果は世代番号で破棄する
- SwiftUIの `ForEach` は安定した写真識別子を使う

### サムネイル

- サムネイル画像はセルローカルの状態に入れる
- `PhotoLibraryService` の公開状態として全サムネイル辞書を更新しない
- 同じ写真と同じサイズのリクエストは重複実行しない
- 画面外に出たセルのリクエストは可能な範囲でキャンセルする
- 一覧ではセルサイズ相当のサムネイルだけを要求し、高品質取得は詳細画面に限定する

## 計測

実機では次のイベント名で、想定外に重い更新が残っていないか確認する。

- `SearchIndexBatch`
- `SearchIndexSave`
- `PublishIndexProgress`
- `FetchFilterCounts`
- `FetchPhotoPage`
- `ExecuteSearch`
- `ThumbnailRequest`
- `ThumbnailResult`
- `ApplyGridSnapshot`

## 守ること

- 写真アプリ内の元写真・元動画は削除、移動、変更しない
- 写真本体、元動画本体、サムネイル本体はSQLiteへ保存しない
- 表示状態、OCR結果、手動分類、メモ、タグは再読み込みで勝手に消さない
- タブ移動やView再生成をユーザーキャンセル扱いにしない

## 実機確認

- 旧JSON移行中に写真画面が白画面にならない
- 進捗カード更新中もグリッドスクロールが固まらない
- 件数チップは横スクロール1行で崩れない
- 検索入力後も200件単位で結果が表示される
- サムネイル取得で画面全体が点滅しない
- 写真タブ、検索タブ、設定タブを移動しても読み込み状態が破綻しない
