# まとめてOCR設計

## 目的

大量写真ライブラリで全件OCRを初期実行しない。ユーザーが必要な範囲を選び、端末状態を見ながら小分けにOCRする。

## 実装済み

- 対象は表示中、スクショのみ、書類写真候補のみ、未OCRのみから選ぶ
- 件数は20/50/100件から選ぶ
- OCR済み、処理中の写真は既定で除外する
- 実行前に安全確認を表示する
- キャンセル可能
- 完了分は保存し、未処理分は未処理のまま残す
- 処理中は完了件数、失敗件数、キャンセル状態を表示する
- 写真タブのOCRエリアは折り畳み可能にし、通常時は1行操作と短い状態表示だけにする
- 進捗詳細、説明文、表示中のOCR結果削除は詳細展開時だけ表示する
- OCR結果削除はしまい箱内のOCR結果だけを対象にし、元写真・元動画、手動分類、不要候補、メモ、タグは変更しない

## メニュー構成

写真タブのOCRメニューでは、小分けOCRと永続ジョブOCRを分けて扱う。`まとめてOCR` ボタンは20/50/100件の `quick` 計画だけを開始し、全数OCRや高精度OCRは開始しない。

- `20件`
- `50件`
- `100件`

全数OCRはOCR詳細内の `全数OCRを管理` から専用確認画面へ進む。

- `現在の絞り込み結果すべて`
- `スマート全件OCR`
- `検索精度をさらに上げる`

全数OCRジョブが `preparing` / `running` / `throttled` / `paused` の間も、`全数OCRを管理` は有効にする。実行中は新規開始ではなく、一時停止、再開、残りの処理を終了する操作を出す。`まとめてOCR` が待機になる場合は、全数OCR実行中であることを短く表示する。

写真画面では全数OCRの `activeSnapshot` がある間、コンパクト進捗カードを表示する。`処理済み / total`、ProgressView、現在phase、Heartbeatを小さく表示し、詳細は `詳細` から確認する。全数OCR中の `まとめてOCR` は競合防止のため無効化し、待機理由を表示する。

全数OCRの進捗カードは写真グリッドとは独立させる。`PhotoGridView` 親Viewは進捗Storeを直接読まず、専用の小さなカードだけが `OCRProgressStore` を購読する。これにより、OCR 1件完了ごとに写真一覧、検索バー、カテゴリチップが再描画されることを避ける。

完了時の表示はコンパクトにする。`処理済み n / total件`、文字あり、文字なし、失敗だけを残し、速度、残り時間、現在phase、Heartbeat、停止警告、操作ボタンは非表示にする。

`20件`、`50件`、`100件` は従来の表示中候補からの小分けOCRである。`現在の絞り込み結果すべて` は、検索語、表示状態、カテゴリ、スクショ細分類を固定した永続ジョブとして扱う。実行中に画面の絞り込み条件が変わっても、ジョブ対象は開始時の計画から変えない。

`スマート全件OCR` はスクショ・書類系を優先して少しずつ進める推奨モードである。`検索精度をさらに上げる` は上級者向けの入口として扱い、非常に時間がかかること、発熱とバッテリー消費が大きいことを開始前に強く表示する。実機でスマート全件OCRの安定性を見ながら高精度化を拡張する。

## OCRExecutionPlan

確認画面、候補件数、端末状態判定、開始可否、ジョブ作成は `OCRExecutionPlan` から決定する。クイックOCRは `small` / `medium` の workload として扱い、対象が少ないのに全数OCR向けの温度判定で止まらないようにする。

設定画面のOCR設定にも同じ内容を表示する。古い固定上限ではなく、クイックOCRは `20 / 50 / 100件` と表示し、絞り込み結果OCR、スマート全数OCR、全数高精度OCRの位置づけを確認できるようにする。

## 安全方針

- OCRは端末内で実行する
- 写真は外部送信しない
- 写真アプリ内の元写真・元動画は削除・変更しない
- 全数OCRは長時間処理のため、thermal fairでは減速し、thermal serious/critical、低電力モード、メモリ警告、空き容量不足では一時停止する
- thermal nominalでも端末負荷を抑えるため短い休みを挟む。thermal fairでは停止ではなく強く減速し、同じ警告を連発しない
- 約10分処理したら短い休憩を入れ、長時間OCRでの発熱を抑える
- iCloud取得を許可している場合、iOSの写真機能がAppleのiCloudから画像を取得することがある
- 同じ発熱警告をToastやAlertで連発しない
- 発熱、低電力モード、空き容量不足、アプリ状態変化では安全側に中断する

## OCR状態

永続ジョブでは、OCR完了を次のように分ける。

- `completedText`: 文字が取得できた。OCR本文を保存し、検索文書へ反映する
- `completedNoText`: OCRは正常完了したが文字が見つからない。失敗ではなく、既定では再処理しない
- `cloudPending`: iCloud上の写真を取得できない、またはオフライン優先のため取得しない。失敗ではなく、iCloud取得許可時に再試行する

この3状態を分けることで、文字なし写真を失敗扱いで何度も処理したり、iCloud待ち写真を失敗として埋もれさせたりしない。

## 今後の改善

- 実機でスマート全件OCRを長時間確認する
- 端末状態が良い場合に限り2並列を検討する
- iCloud待ち件数が多い場合の確認シートをより分かりやすくする
- OCR済みを除く導線を検索タブ側にも広げる
- 全件高精度OCRの画質設定と再OCR条件を実機結果から詰める
- 1件完了ごとにSQLiteへ保存し、進捗UIは最大で毎秒2回程度に間引く
- OCR画像は長辺1600〜2048px程度に抑え、詳細画面だけ高品質画像を検討する
- OCR進捗は専用ジョブ状態として更新し、PhotoGrid全体や件数チップを毎回更新しない
- 全数OCRの進捗は `OCRProgressStore` の値型スナップショットとして表示し、写真グリッドの配列更新から分離する
- Heartbeatでワーカー生存状態を確認し、30秒以上進捗更新がない場合は進捗カードに停止可能性を表示する

## やってはいけないこと

- 全数OCRを単純な一括ループで実装する
- 全件OCR対象をSwiftUI配列として公開する
- 全写真の画像本体をメモリに保持する
- OCR進捗ごとに写真一覧全体を再描画する
- iCloud写真を無断で大量取得する
- 写真アプリ内の元写真・元動画を削除・変更する
