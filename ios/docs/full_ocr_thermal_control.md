# 全数OCRの発熱制御

## 目的

全数OCRは長時間動くため、短時間で完了させるよりも、中断しても少しずつ確実に進むことを優先する。

OCR結果は端末内に保存し、元写真・元動画は削除・変更しない。

## 状態別の扱い

- `nominal`: 通常実行。ただし長時間OCRでは1件ごとに短い休みを入れ、端末負荷を抑える
- `fair`: 停止ではなく強く減速する。カード内に「端末の温度を見ながらゆっくり処理しています」と1つだけ表示する
- `serious`: 現在処理中の1件を終えた後、一時停止して続きから再開できる状態にする
- `critical`: 端末保護を優先し、安全停止する。完了済みOCR結果は残す

同じ発熱理由をToastやAlertで連発しない。発熱状態は進捗カード内の状態表示にまとめる。

## Duty cycle

初期値は1並列で、次の目安を使う。

- スマート全数OCR nominal: 35〜40%
- スマート全数OCR fair: 15〜20%
- 全数高精度OCR nominal: 25〜30%
- 全数高精度OCR fair: 10〜15%

充電中は発熱しやすいため、さらに10〜20%程度ペースを落とす。

## 休憩

約10分処理したら2分程度の休憩を入れる。休憩中も進捗は保存済みで、元写真・元動画には触れない。

## UI

完了時は速度、残り時間、現在phase、Heartbeat警告を消し、処理済み、文字あり、文字なし、失敗だけを表示する。

発熱による一時停止時は、完了済みのOCR結果が保存されていること、端末が冷めたら続きから再開できることを表示する。

## 実機で見る点

- thermal fairで細かい中断表示が連発しない
- serious/criticalでは安全側に一時停止または停止する
- 発熱中でも写真一覧、検索、設定の操作が固まらない
- 完了済みOCR結果が残る
- 元写真・元動画が写真アプリ側に残る
