# 将来の画像分類改善計画

## 目的

花・植物、桜・紅葉、建物・街並み、神社・寺・史跡、芸術品・展示、食べ物、ペット・動物、人物写真などは、OCRテキストだけでは判定しにくい。現段階では低い信頼度の候補表示に留め、将来的に端末内処理で精度を上げる。

## 現在の制約

- 写真本体は外部送信しない
- 外部APIは使わない
- 写真本体やサムネイル本体をアプリDBに保存しない
- 写真アプリ側のアルバムやフォルダは作らない
- 分類はしまい箱内の仮想フォルダとして扱う
- すべて「候補」として表示し、断定しない

## 現在の分類

現在は次の軽量情報から候補分類する。

- PhotoKitメディア種別
- PhotoKitメディアサブタイプ
- スクリーンショット判定
- 画像サイズ比率
- OCRテキスト

OCRテキストに花、植物、神社、展示、料理、犬、猫などの語句がある場合は候補にする。ただし、写真内容を直接見ているわけではないため信頼度は低めにする。

## 将来案

1. Visionの画像特徴量を端末内で取得する
2. Core MLの端末内モデルを検討する
3. 分類結果だけを `PhotoIndexRecord` に保存する
4. 写真本体や特徴量の原本は保存しない方針を維持する
5. ユーザーが分類を修正できる仕組みを追加する

## 保存方針

保存するのは検索用メタデータだけにする。

- `localIdentifier`
- `inferredCategory`
- `categoryConfidence`
- `categoryReason`
- `categoryUpdatedAt`

将来、画像特徴量を使う場合も、保存するのは最終的な候補カテゴリと信頼度に限定する。写真本体は保存しない。

## 実機確認ポイント

- 一般写真が過剰に特定カテゴリへ寄りすぎないこと
- 低信頼度の候補表示として自然に見えること
- 検索やカテゴリチップが3万枚規模で重くならないこと
- iCloud写真の取得を不要に増やさないこと
- バッテリー、発熱、保存容量への影響が小さいこと

## 次にやること

- 実機写真で誤分類例を集める
- OCR辞書で改善できるものと画像特徴量が必要なものを分ける
- 端末内で完結する小さな分類試作を作る
- SQLite/FTS移行後に分類検索の速度を再確認する
