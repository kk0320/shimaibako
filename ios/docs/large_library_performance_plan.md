# 大量写真ライブラリ性能改善計画

## 目的

28,000件から30,000件規模の写真ライブラリでも、白画面、永久読み込み、タップ不能を避ける。写真一覧を使える状態と、OCRや分類などの解析完了状態を分離する。

## 今回の変更

- `PhotoIndexStoring` の実装として `SQLitePhotoIndexStore` を追加した
- 通常起動時の `PhotoIndexService` はSQLiteを使う
- `photo_index.json` は旧データ移行元、互換バックアップ、障害解析用スナップショットとして残す
- SQLiteが空の場合だけ、旧JSONから200件ずつ移行する
- 既存SQLiteの検索文書バックフィルも200件単位にする
- SQLite移行中は200件単位の進捗だけを専用ストアへ出し、写真グリッド全体を再描画しない
- 一覧は初期200件だけSwiftUIへ渡し、「さらに表示」で200件ずつ増やす
- 検索入力は350ms debounce後に反映する
- 検索結果、表示状態、カテゴリ、スクショ細分類をSQLiteのIDページングで取得する
- 表示状態、カテゴリ、スクショ細分類の件数はSQLite集計を `FilterCountsSnapshot` として一括更新する
- 集計中の件数は `0` ではなく `--` として表示し、未計算とゼロ件を区別する
- カテゴリの「すべて」は現在の表示状態スコープで数える
- 再表示時の全件 `rebuild` を避け、表示中ページの範囲だけに制限する
- サムネイルはセル単位で保持し、同じ画像サイズの重複リクエストを抑える
- 画面外へ出たセルのサムネイル要求は可能な範囲でキャンセルする
- フル/全件読み込み時もSwiftUIへ28,000件規模の配列を渡さず、表示用に保持する `PhotoAsset` は最大500件程度に制限する
- SQLiteページIDから必要な写真だけ `PHAsset` として復元する
- メモリ警告を受けた場合はサムネイルキャッシュを破棄し、読み込みを一時停止して続きから再開できる状態にする
- まとめてOCR件数を20/50/100件から選べるようにした
- 読み込み状態リセットは復旧オプションへ移し、確認ダイアログと二重実行防止を入れた
- 詳細画面の不要候補/通常表示操作には軽量な「元に戻す」を追加した

## データの流れ

```text
PhotoKit
  ↓
PhotoLibraryService
  ↓ 最新バッチ100件 / 表示保持500件程度
PhotoIndexService
  ↓
SQLitePhotoIndexStore
  ↓
PhotoGridView には表示に必要なIDページだけ渡す
```

写真本体、サムネイル本体、元動画本体はSQLiteへ保存しない。保存するのは `localIdentifier` を主キーにした検索用メタデータ、分類、OCRテキスト、メモ、タグ、表示状態である。

## 更新経路の分離

大量ライブラリでは、1つの `@Published` 更新にすべてを乗せると、インデックス進捗、件数チップ、写真グリッド、サムネイルが互いに巻き込まれて再描画される。今回の変更では次の4経路を分ける。

- 検索インデックス進捗: `IndexProgressStore` にだけ流し、写真グリッド全体を更新しない
- 件数チップ: `FilterCountsSnapshot` をSQLite集計から作り、一括で差し替える
- 写真グリッド: SQLiteから `localIdentifier` のページを取得し、200件単位で適用する
- サムネイル: セル単位のローカル状態へ配信し、重複要求と画面外要求を抑える
- OCRジョブ進捗: `OCRJob` / `OCRJobItem` の専用状態として流し、写真グリッドや件数チップの全更新につなげない

この分離により、旧JSON移行や検索インデックス作成の進捗更新が、28,000件規模の写真一覧を毎回作り直す経路を避ける。

## 将来のフルOCR方針

`coredata-performance-refactor` ブランチではフルOCRを実装しない。大量写真向けの永続OCRジョブは、このブランチを `main` へmergeしたあと、`feature/persistent-full-ocr` ブランチで実装する。

将来のOCRメニューは段階的にする。

- `20件`
- `50件`
- `100件`
- `現在の絞り込み結果すべて`
- `スマート全件OCR`
- `全件高精度OCR`

`スマート全件OCR` は、スクショ、書類、領収、名刺、看板、白板など検索価値が高い候補から優先する。一般写真とiCloud取得が必要な写真は後回しにする。

`全件高精度OCR` は、実機でスマート全件OCRの安定性を確認した後に検討する重いモードである。初期実装では追加しない。

永続ジョブでは、OCR結果を次のように分ける。

- `completedText`: OCR本文あり。検索文書を更新する
- `completedNoText`: OCR正常完了だが文字なし。失敗扱いにしない
- `cloudPending`: iCloud取得待ち。失敗扱いにせず再試行対象にする

OCRは初期1並列にする。端末状態が良く、実機でUIが固まらないことを確認できた場合のみ2並列を検討する。低電力モード、発熱、メモリ警告、空き容量不足では一時停止する。

OCR結果保存と `SearchDocument` 更新は1件単位で行う。OCR進捗ごとに `PhotoGridView`、件数チップ、カテゴリチップを全更新しない。

## 計測ポイント

実機ログでは、次のイベント名で重い箇所を確認する。

- `SearchIndexBatch`
- `SearchIndexSave`
- `PublishIndexProgress`
- `FetchFilterCounts`
- `FetchPhotoPage`
- `ExecuteSearch`
- `ThumbnailRequest`
- `ThumbnailResult`
- `ApplyGridSnapshot`
- `photoFetch`
- `sqliteBatchInsert`
- `finalization`

目的は、進捗通知やサムネイル結果が写真グリッド全体の再生成へ波及していないかを実機で見ることである。

## UI方針

- 初期表示は200件
- 残りはユーザー操作で200件ずつ表示
- SwiftUIのセル生成数を抑える
- カテゴリチップとスクショ細分類チップは1行固定の横スクロールにする
- スマホ幅ではカテゴリチップ群を折り畳み可能にし、写真一覧の表示領域を優先する
- チップ列は横方向専用のスクロールにし、縦方向のバウンドや位置ずれを抑える
- 長いチップ名は行高を変えずに収める
- 件数の集計中は `--` を表示し、ゼロ件として見せない
- OCR操作行は1行表示を基本とし、件数選択と実行ボタンで縦幅を増やしすぎない
- タブ移動やView再描画で長時間読み込みジョブをユーザーキャンセル扱いにしない
- 進捗や復旧UIでは「元写真・元動画は削除・変更しません」と明記する

## 残る課題

- `PhotoLibraryService.assets` は表示保持用に残すが、大規模モードでは最大500件程度に制限する
- 写真グリッドのIDページング後は、ページIDだけをPhotoKitから復元する
- 真のPhotoKitカーソル型ロードや差分監視は次段階で検討する
- `PHPhotoLibraryChangeObserver` による差分反映は未実装
- 非表示/整理済みの直接操作UIを追加する場合は、その操作にもUndoを付ける必要がある
- 実機ログでサムネイル要求の重複が残る場合は、プリヒート範囲の調整を検討する
- 永続OCRジョブ、現在の絞り込み結果すべて、スマート全件OCRは未実装

## 実機で見る点

- 2,000件モードでタブ切替後も写真画面が白画面にならない
- フル読み込み済み状態で先頭200件が先に表示される
- 「さらに表示」で段階表示できる
- 件数チップの表示やスクロールで固まらない
- 集計中に `通常 0` のような誤解を招く表示にならず、`--` で準備中と分かる
- カテゴリの「すべて」が現在の表示状態と同じスコープで表示される
- カテゴリチップ群を閉じた状態で写真一覧の操作領域が広くなる
- カテゴリチップ列を斜めにドラッグしても上下へずれない
- OCR操作行の `20件` と `まとめてOCR` がスマホ幅で折り返されない
- 27,000件以降の終盤で、写真グリッドへ全件配列が流れず、進捗カードが更新され続ける
- メモリ警告が出た場合に `端末のメモリ不足により一時停止しました` として復旧できる
- 検索入力中に固まらない
- 検索結果も200件単位で追加表示される
- OCR件数を20/50/100から選べる
- 読み込み状態リセットは確認ダイアログなしに実行されない
- 不要候補/通常表示のUndoが機能する
- 元写真・元動画が写真アプリ側に残る
